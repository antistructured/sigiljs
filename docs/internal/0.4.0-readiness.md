Date: 2026-06-21
Package: `@weipertda/sigiljs`
Release theme: Public API stabilization

## 3. Experimental APIs

The following symbols are exported but intentionally marked experimental in 0.4.0:

- `httpContract()`
- `contract.toFormConstraints()`

Both are covered by `docs/experimental.md`. They remain available but are not implied as stable boundaries for package extraction.

## 4. Internal APIs

Internal implementation details are not documented as public API. Examples include:

- Parser helpers
- Compiler internals
- AST registry functions
- Projection helper implementations
- Error factory internals

These are implemented in `src/core/`, `src/parser/`, `src/compiler/`, and `src/projections/`, but they are not listed in `docs/api.md` as public entry points.

A user should not depend on internal paths between patch releases.

## 5. Projection stability status

Projection APIs are frozen for 0.4.0:

- `describe()` returns a stable plain description model
- `toJSONSchema()` projects deterministic JSON Schema-like output
- `toTypeScript(name)` emits deterministic TypeScript declarations
- `toOpenAPI()` projects deterministic OpenAPI-compatible schemas

The freeze contract is documented in `docs/projections.md` and `docs/internal/projection-api-freeze.md`.

Outputs are snapshot-tested. Behavior is regression-protected.

## 6. Alias policy

Public aliases:

- `sigil()` is the preferred object-API entry point
- `Sigil`\`template\` is the template API
- `S` and `T` are legacy aliases for `Sigil`

New code should use `sigil()` or `Sigil` directly. `S` / `T` are preserved for compatibility but documented as legacy in `docs/sigils.md`.

## 7. httpContract status

`httpContract()` is **experimental**.

Findings from `docs/internal/http-contract-boundary-review.md`:

- Exported and documented
- Framework-neutral
- Has tests
- Supports request and response contracts
- Uses `parse`/`safeParse` internally
- Output shape is stable today, but the boundary is not yet validated across real adoption

Decision: keep exported, do not treat as stable in README or guides, do not imply `@sigil/http` extraction in current release materials.

## 8. Known limitations

- Forms projection (`toFormConstraints()`) is experimental and may change before 1.0.0
- `httpContract()` is experimental and not a committed stable boundary
- Package is not yet split into scoped subpackages
- Some internal helpers are close to the surface but intentionally excluded from public docs
- Existing lint warnings in `src/sigil.js` and `src/projections/typescript.js` are pre-existing and unrelated to this stabilization block

## 9. Release recommendation

Ready for 0.4.0

The public API surface is:
- enumerated
- tested
- documented
- classified as stable, experimental, or internal

No core system was rewritten. No package was split. The test suite passes with deterministic outputs.

## Suggested next block after 0.4.0

After 0.4.0, the recommended focus is:

- Evaluate `httpContract()` and `toFormConstraints()` for stable promotion based on real usage
- Begin package-extraction research only if adoption data justifies it
- Tighten deprecation policy for legacy aliases `S` and `T`
